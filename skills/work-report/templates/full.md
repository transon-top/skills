# full 模式模板

按大类分组，大类下按功能域分节。面向技术领导，叙述完整连贯，说明做了什么、为什么做、影响是什么。

## 大类与内容要求

- **新功能开发**：新增的用户可见能力、业务模块、API 端点。需说明功能目标和用户价值。
- **问题修复**：已修复的线上缺陷、逻辑错误、数据一致性问题。需说明问题现象和根因。
- **UI/UX 优化**：界面布局调整、交互流程改进、暗黑模式等视觉适配。需说明改进前后的体验差异。
- **代码重构与架构优化**：模块拆分、文件重组、技术债务清理、错误边界等容错机制。需说明重构动机和收益。
- **性能优化**：加载速度、渲染效率、资源占用、网络请求的改进。需说明优化前后的指标变化。
- **基础设施与工程效能**：依赖升级、CI/CD 调整、构建配置、代理配置、调试工具。需说明变更对开发流程的影响。

只输出有提交的大类，无提交的大类不出现。

## 输出格式

功能域按大类排序：新功能开发 → 问题修复 → UI/UX 优化 → 代码重构与架构优化 → 性能优化 → 基础设施与工程效能。同大类内按提交数量降序排列。

每个功能域包含：

- **背景**：此阶段该功能域要解决的问题或整体目标（1-2 句）
- **开发明细**：逐条叙述，串联相关提交为连贯描述，每条是一段完整的话而非提交信息的简单翻译

## 约束

- 功能域名称保持稳定：同一功能域在多个提交中出现时合并为一节，不重复拆开
- 背景体现业务意图或问题根因，不描述技术实现
- 跨提交的关联逻辑串联为连贯叙述，而非孤立罗列
