# 架构逆向工程

对目标项目仓库进行系统性逆向分析，产出三份结构化文档。

**分析目标**：用户指定的目录，未指定则默认为当前工作目录。

**输出路径**：用户可指定输出根路径（如 `-o /data/projects`），未指定则默认使用当前工作目录下的 `docs/`。三份文档统一写入 `<输出根路径>/<project-name>/` 文件夹：

- `<project-name>/feature-list.md`
- `<project-name>/user-stories.md`
- `<project-name>/resume-entry.md`

`project-name` 从被分析项目的 `package.json` 的 `name` 字段或项目目录名提取，转为 kebab-case。**第一步先创建输出文件夹**，再将三份文档写入其中。

## 模板

每个输出文件格式由以下模板定义，生成前先读取对应模板：

- [`templates/FEATURE_LIST.md`](./templates/FEATURE_LIST.md)
- [`templates/USER_STORIES.md`](./templates/USER_STORIES.md)
- [`templates/RESUME_ENTRY.md`](./templates/RESUME_ENTRY.md)

## 步骤

### 第〇步：发现辅助工具

在进入信息采集前，先检查被分析项目中是否有辅助代码阅读的工具：

1. 读取被分析项目的 `CLAUDE.md`（或 `.claude/CLAUDE.md`），查找其中注册的代码分析工具、MCP 工具、或自定义脚本（如 codegraph、zread、repo-structure 等）
2. 检查当前会话可用的 MCP 工具中是否有代码结构分析类工具（如 `mcp__zread__get_repo_structure`、`mcp__zread__search_doc` 等）
3. 检查项目中是否有 `.codegraph/`、`graph.json`、`dependency-graph.json` 等预生成的代码图谱文件

**发现工具则优先使用** — 用工具获取项目结构、模块依赖关系、文件列表，大幅减少手动 grep 和逐个文件读取。

**完成标准**：已确认有哪些辅助工具可用（或确认无可用工具），工具清单已记录。有工具时，第一步中已由工具覆盖的项目结构、文件列表等项可跳过不重复扫描。

### 第一步：信息采集

第〇步已由工具获取的信息（项目结构、文件列表、模块依赖）直接复用，仅补充工具未覆盖的部分。按以下顺序通读剩余关键文件，构建项目的完整心智模型：

1. `package.json` — 依赖项、脚本、workspace 配置
2. `README.md` — 项目定位、安装说明、架构概述
3. 配置文件 — `tsconfig.json`、`next.config.*`、`vite.config.*`、`turbo.json` 等
4. 顶层目录树 — monorepo 包结构、应用目录、共享库
5. 路由定义 — `app/`（Next.js）、`src/routes/`（Express）、路由配置文件
6. 核心源码目录 — 组件、hooks、服务层、数据模型、工具函数
7. API 层 — service 文件、HTTP 请求封装、API 路由处理
8. 状态管理 — store、context、查询缓存

**完成标准**：能逐条说出：项目做什么、为谁做、技术栈、顶层模块、每个路由/页面、UI 到后端的数据流路径。

### 第二步：Git 历史分析（用户故事阶段排序的前置步骤）

从 git 提交历史中提取功能的演进时序，为第三步用户故事的阶段划分提供数据依据。

执行以下命令并记录结果：

```bash
git log --oneline --reverse        # 提交时间线，从最早到最新
git log --oneline --no-merges      # 排除合并提交后的功能提交
git log --format="%h %s" --diff-filter=A -- "*.ts" "*.tsx" "*.js" "*.vue"  # 文件首次创建记录
git tag --sort=creatordate         # 版本标签时间线
```

分析要点：

- 从提交信息中识别功能引入的先后顺序——先有的模块是什么，后加的是什么
- 识别提交信息中的功能关键词（feat、add、refactor、模块名）并按时序排列
- 文件首次创建时间对应功能的诞生顺序
- 版本标签标记功能分批上线的节点

**输出**：一张功能诞生时序表，列出每个功能模块首次出现的时间点和对应的提交。这张表直接决定第三步用户故事的阶段排列。

### 第三步：生成产品功能清单

从扫描结果中提取所有功能模块。规则：

- 按业务领域分组，不按技术分层
- 子功能必须是用户可感知的行为，非实现细节
- 元数据块（技术形态、开发周期、代码规模）从 git log 和 tokei/cloc 统计中提取，不估算
- 末尾附 ASCII 架构全景图和关键目录结构

**完成标准**：扫描发现的所有路由、组件组、API 端点均在功能清单中有对应条目。逐一核对，零遗漏。

### 第四步：生成用户故事地图

从功能清单推导用户故事。**阶段排序以第二步的 git 时序分析结果为权威依据**——功能先出现的排在前面，后出现的排在后面。按用户旅程时间线组织：发现与上手 → 核心工作流 → 配置管理 → 导出与分享 → 账户管理。

其他规则：

- 故事格式：`US-NN 作为<角色>，我希望<目标>` — 角色必须是项目实际服务的用户类型
- 验收标准：一句可测试的话，描述可观察的外部行为
- 优先级：P0 = 缺失则核心体验不可用；P1 = 重要但不阻塞；P2 = 锦上添花
- 末尾附角色权限表，覆盖所有故事中出现的角色

**完成标准**：第三步每个功能模块至少映射到一个用户故事；每个故事有可验证的验收标准；阶段顺序与 git 时序分析结果一致；角色表无遗漏。

### 第五步：生成简历项目经历

综合信息采集和功能清单，输出可直接用于简历的项目描述。四个版本：

**完整版（STAR-L）**：

- S（背景）— 项目解决的问题空间
- T（任务）— 开发者的具体职责维度
- A（行动）— 技术决策及理由：做了什么、怎么做、为什么
- R（结果）— 量化指标：提交数、代码行数、文件数、模块数、性能数据
- L（学习）— 3-6 条可迁移的认知洞察

**精简版**：3-4 条要点，每条包含模块名 + 动作 + 关键技术 + 结果，最多 2 行。

**一句话版**：单句，角色 + 核心工作 + 技术栈 + 规模，最长 80 字。

**面试追问**：3-5 组问答，针对架构权衡、技术创新点、性能瓶颈突破等有深度的主题。

**完成标准**：四个版本齐全；Action 部分覆盖第三步每个模块；Result 均为量化数据；面试问答聚焦项目独有的技术深度，不出现功能清单可回答的浅层问题。

## 约束

- **工具优先于手动搜索** — 第〇步发现的辅助工具（codegraph、MCP 结构分析工具等）必须在第一步信息采集中实际使用，能由工具获取的信息不靠逐个文件读取。
- **输出文件夹先于内容** — 生成任何文件前，先创建 `<输出根路径>/<project-name>/` 文件夹。三份文档写入同一文件夹。
- **先读模板再生成** — 严格匹配模板的标题层级、元数据字段和章节顺序。
- **git 数据优先** — 提交数、代码行数、功能时序均从 git 提取，不靠记忆或估算。
- **功能清单求全，简历经历求精** — 功能清单是完整规格说明；简历只取最具技术含金量的工作。
- **用户故事的用户视角** — 描述用户能做什么，不描述系统内部做什么。
